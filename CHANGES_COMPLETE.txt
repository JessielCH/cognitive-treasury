🔄 CAMBIOS PRINCIPALES
═══════════════════════════════════════════════════════════════════════════════

CAMBIO #1: URLs Dinámicas en Frontend
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Problema:
  Backend en localhost:8000 en desarrollo, pero ¿en producción?
  Las URLs estaban hardcodeadas en el código.

Solución:
  const API_URL = import.meta.env.VITE_API_URL || "http://localhost:8000";
  
  En desarrollo: VITE_API_URL=http://localhost:8000 (.env.development)
  En producción: VITE_API_URL=https://your-railway-app.railway.app (Railway)

Beneficio:
  ✅ Un mismo código sirve para dev y prod
  ✅ Railway inyecta la URL correcta automáticamente
  ✅ Sin rebuilds del frontend por cambios de URL


CAMBIO #2: Persistencia Configurable de RLHF Weights
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Problema:
  Los pesos RLHF se guardaban en api/rlhf_weights.json
  En Railway, cada deploy recrea el filesystem → pesos se pierden
  No hay forma de persistir datos

Solución:
  WEIGHTS_DIR = os.getenv("WEIGHTS_STORAGE_PATH", 
                          os.path.join("api", "data"))
  os.makedirs(WEIGHTS_DIR, exist_ok=True)
  
  En Railway: WEIGHTS_STORAGE_PATH=/app/api/data (volumen montado)

Beneficio:
  ✅ Path configurable para cualquier ambiente
  ✅ Automáticamente crea directorios si no existen
  ✅ Preparado para futuro con PostgreSQL/Redis
  ✅ Con volumen en Railway, pesos persisten entre deploys


CAMBIO #3: Docker Multi-Stage Build
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Problema:
  Necesitamos Node.js (React) + Python (FastAPI) en la misma imagen
  ¿Cómo empaquetarlos juntos sin duplicar dependencias?

Solución:
  Stage 1 (Node 20): Compila React → genera /dist
  Stage 2 (Python 3.11): Instala deps Python + copia /dist
  Resultado: Una sola imagen con todo

Beneficio:
  ✅ Una sola imagen de Docker
  ✅ Sin duplicar Node.js en la final
  ✅ Imagen optimizada (~500MB vs ~1GB)
  ✅ Sirve frontend + backend desde FastAPI
